## (historical) STEP 2 plan — remove config-on-the-handle (one coupled green pass)
Goal end-state (maintainer-confirmed): config lives ONLY at Jdbi level (immutable/shared) + statement level (defines overlay + per-execution state). The handle is connection + transaction only and holds NO config. Then `StatementTemplate.with(handle)` parenting from the template's (Jdbi-level) config is correct by construction (the B1 "handle config ignored" concern disappears — see "Locked decisions").

Removing the API breaks all callers until migrated, so (a) and (b) land together as one green commit.

**(a) Migrate the ~40 test callers of the scoped-open API to `ext.openWithConfig(b -> b.configure(...))`.**
- The scoped overloads are `Jdbi.open(Consumer<ConfigRegistry>)`, and scoped `withHandle`/`useHandle`/`inTransaction`/`useTransaction(Consumer<ConfigRegistry>, callback)`.
- Audit result: EVERY caller is a test. None are production. Each maps to Jdbi-level config (most), a `ScopedValue`-read config object (per-request logger), or per-statement config. Nothing is stranded.
- Find them: `grep -rnE '\.open\([a-z]*[ ]*->|\.(useHandle|withHandle|inTransaction|useTransaction)\([^,]*->[^,]*,' --include='*.java' . | grep -v /Jdbi.java | grep -viE 'withHandle\(h ->|useHandle\(h ->|open\(\)'`
- Migration shape: `X.getJdbi().open(cfg -> cfg.configure(K.class, op))` → `X.openWithConfig(cfg -> cfg.configure(K.class, op))`. NOTE: `Jdbi.Builder.configure(Class, UnaryOperator)` has the SAME signature as `ConfigRegistry.configure`, so the lambda body is unchanged — only the receiver call changes (`getJdbi().open(` → `openWithConfig(` on the extension). Where a local `jdbi` var is used, switch to the extension reference (the site has one, e.g. `h2Extension`/`sqliteExtension`).
- Two-arg `useHandle/withHandle/inTransaction/useTransaction(scope, callback)` sites (~8: `EnumsConfigTest` x3, `QualifiedEnumArgumentTest` x3, `TestMapperInit` x2) need restructuring to `try (Handle h = ext.openWithConfig(cfg -> ...)) { <callback body> }` (and `return h.inTransaction(...)` for the transaction variants).
- Known caller files (verify with the grep; list may be incomplete): TestClosingHandle, EnumsConfigTest, HandleExceptionTest, transaction/TestTransactions, extension/TestExtensionCustomizers, argument/QualifiedEnumArgumentTest, argument/TestArgumentFactory, argument/TestPreparedArguments, qualifier/TestCustomQualifier (many), mapper/TestMapperInit, mapper/reflect/BeanMapperNestedTest, mapper/reflect/FieldMapperTest, mapper/reflect/ConstructorMapperTest, collector/CollectorsTest, statement/TestSqlLoggerAttributesAndBinding, statement/TestSlf4JSqlLogger, statement/TestSqlLoggerToString, statement/TestTimingCollector, statement/LeakTest, statement/TestScript.

**(b) Remove the scoped-open API.** In `Jdbi.java`: delete the 5 scoped overloads (`open(Consumer)`, `withHandle(Consumer,..)`, `useHandle(Consumer,..)`, `inTransaction(Consumer,..)`, `useTransaction(Consumer,..)`) and the `NO_CONFIG_SCOPE` constant. In `Handle.java` ctor: remove the `Consumer<ConfigRegistry> configScope` parameter and its `configScope.accept(handleConfig)` call; the handle config becomes just `jdbi.getConfig().createChild()` (unforked, never mutated). Fix the non-scoped `open()`/`withHandle`/etc. that delegated through the scoped ctor path.

**(c) Finalize `customizeHandleConfig` — remove the mutate-handle-config SPI.** After the spike it is already a non-forking side-effect, but it still HANDS plugins a mutable `ConfigRegistry` (foot-gun). Change the SPI parameter from `ConfigRegistry` to `ConfigView` (read-only) so plugins can read Jdbi-level config + do connection side-effects but cannot mutate handle config. `PostgresPlugin.customizeHandleConfig` uses only `config.get(...)`, which works on `ConfigView`. Update `Jdbi.customizeHandleConfig` + `JdbiPlugin.customizeHandleConfig` signatures. (Renaming it — it no longer "customizes config" — is optional cosmetic follow-up; there is already a separate `customizeConnection(Connection)`.)

**(d) Docs.** Add an upgrade-guide section (`docs/src/adoc/index.adoc`, the `== Upgrading to Jdbi 4` area) documenting the three replacements for the removed handle-config scope: (1) per-tenant/bounded → `jdbi.toBuilder()....build()` (one derived Jdbi); (2) per-request ambient context (request id, tenant) → a Jdbi-level config object (e.g. a `SqlLogger`) that reads a `ScopedValue` (final in Java 25; structured-concurrency-safe; `ThreadLocal` fallback) — include the short `RequestTaggingLogger` example from the conversation; (3) this-one-statement → `stmt.configure(...)`/`define(...)`. Also note `Jdbi.open(Consumer)`/scoped `withHandle` removal.

**(e) STEP 3 after a/b/c green:** add the `with(handle)` handle-visibility regression test (was task #4) — now that the handle carries no config, a template built from `jdbi` and run on any handle sees the same (Jdbi-level) config; also assert a Postgres handle's config stays an unforked child of the root. Then `git rm STEP2-HANDOFF.md` scope is done.

## Locked design decisions (confirmed with the maintainer this session)
- Defines are per-render statement state, NOT config (done). `RenderContext(config, overlay)` already existed; now wired.
- `with(handle)` must NOT parent the statement from the raw handle config — that was tried and REVERTED because it discards SQL Object's CONFIGURE-baked customizers (`@AllowUnusedBindings`/`@Timestamped`/`@MaxRows`; 6 failures). For SQL Object the template is built FROM the handle's config (`instanceConfig <- handleSupplier.getConfig()`), so template-config parenting already includes per-handle config. The correct fix is the no-handle-config end-state (STEP 2), after which template-config parenting is right by construction. Do NOT re-attempt plain handle-parenting.
- `open(scope)` is removed (not reframed) because its use cases are all better served by toBuilder / ScopedValue / per-statement config.
- Plugin config precedence (task #5 below): maintainer chose CALLER CONFIG WINS.

## (historical) Remaining tracked review findings (from the 6-agent review; were harness tasks #5–#12)
- **#5 Plugin precedence — DECIDED: caller wins. IMPLEMENT.** The v4 builder queues plugins and runs `configure()` at `build()` (last), so a plugin overrides caller config registered earlier — an inversion from 3.x. Fix: apply queued plugins BEFORE the caller's builder registrations (drain at the start of `build()`, or apply inline at `installPlugin`) so caller config prepends over plugin config. Add a regression test. Surfaces in guice `InternalGuiceJdbiCustomizer` and spring `JdbiFactoryBean`. Also: `JdbiFactoryBean` silently dropped `setAutoInstallPlugins` (loud `NotWritablePropertyException` for existing beans) — confirm intended + note in upgrade guide.
- **#6 Harden `DefaultJdbiCache` against a concurrent throwing loader.** `doGet` fix removes the node on loader failure but a thread already waiting on the node's monitor re-checks `!isDone()` (still true), re-runs the loader, and on success completes + LRU-enqueues a node no longer in the cache map — orphan that later evicts the live entry via one-arg `cache.remove(key)` and inflates `expungeQueue.size`. Latent (needs a non-deterministic loader; jdbi loaders are pure). Fix: on loader failure complete the future exceptionally so a waiter can't resurrect a removed node.
- **#7 Narrow `StatementTemplate` ctor `catch (RuntimeException ignored)`** (StatementTemplate.java ~line 70). It swallows ALL render/parse failures as "SQL depends on per-exec attributes", masking genuine template/parser faults and silently disabling the render-once fast path. Narrow to the expected signal, or at least don't treat arbitrary programming errors as expected control flow.
- **#8 `JdbiConfig` SPI javadoc still blesses in-place mutation** ("thread-safe for access and caching, but not necessarily for reconfiguration"). Under the shared-immutable model a third-party mutable config's in-place mutator, via `configure()` on an unforked scope, mutates the shared ancestor (no defensive copy in `configure`/`install`; `fork()` copies by reference). Core configs are all immutable so this only bites third parties. State the immutability requirement; drop the stale sentence.
- **#9 Improve diagnostics/docs for third-party CONFIGURE-phase customizers.** `phaseFor` defaults any unmarked type/method-level customizer to CONFIGURE (baked once). A pre-4.0 third-party method/type customizer that touches the live statement dies with a cryptic `UnsupportedOperationException` from `ConfigureStatement`; one that mutates config per-invocation without `ConfigMutating` silently bakes once. Intended v4 break (documented), but improve the failure message to name the fix (declare `StatementScoped`/`ConfigMutating`), and note that a CONFIGURE customizer registering a STATEFUL inner `StatementCustomizer` now shares one instance across concurrent executions.
- **#10 Fix inaccurate comments.** (1) `ConfigRegistry.readAs` javadoc says views are "scoped to this registry ... never observed by another", but an unforked child shares the parent's memoized view — resolvers ARE shared across threads and must be thread-safe; fix wording. (2) The fields comment justifies safety with "safely published, then frozen" but the root is not frozen (lazy `get()`/`readAs` writes after publication); the real basis is the ConcurrentHashMap maps.
- **#11 `setFetchSize`/`setMaxRows`/`setMaxFieldSize`/`setQueryTimeout` fork config.** They route through `configure()->install()->fork()`. After the defines-overlay redesign the SQL-reuse gate no longer keys on fork state (it keys on the defines overlay being empty), so verify these no longer defeat the template SQL cache; if they still fork config unnecessarily, move this per-statement state off config (statement state) per the design doc's field taxonomy. Add a test.
- **#12 Audit and remove now-useless caches in core + sqlobject.** With templates owning render+parse and per-instance immutable-config resolvers, several legacy caches may be dead (SqlStatements template/parse cache, resolver/metadata caches, ConfigCaches/JdbiCaches). Enumerate every cache; remove those templates + immutable config made redundant, keeping only a decoupled pure SQL-text->parsed memoization for naive repeated classic usage IF measurement justifies it (design doc "Caches unwind" / phase 4). Validate with the benchmark suite that removal doesn't regress the classic path.
